# Storage layout

Artifact storage is content-addressed and evidence-oriented. Package names,
filenames, URLs, archive paths, and registry metadata are hostile input and MUST
NOT be concatenated into filesystem paths or object keys. All storage locations
are produced by the storage URI builder and are safe for the local filesystem and
future S3-compatible object stores.

## URI principles

- URI schemes are explicit: `fs://` for local filesystem-backed storage and
  `s3://` for future object storage.
- Path components are generated from normalized identity fields and cryptographic
  digests only.
- Raw package names, archive filenames, download URLs, maintainer names, and
  archive entry paths are stored as metadata, never as parent directories.
- The URI builder rejects absolute paths, traversal (`..`), path separators in
  opaque IDs, control characters, empty components, and malformed digests.
- Callers pass identity and digest values to the builder; callers do not
  hand-concatenate artifact paths.

## Raw artifact paths

Raw downloaded artifacts are stored only after hash verification has produced a
local SHA-256 digest. The canonical raw artifact URI shape is:

```text
fs://artifacts/raw/{ecosystem}/{package_digest}/{version_digest}/{artifact_sha256}
```

Where:

- `ecosystem` is one of the supported canonical ecosystems, such as `npm` or
  `pypi`.
- `package_digest` is `sha256(normalized package identity key)` encoded as
  lowercase hexadecimal.
- `version_digest` is `sha256(raw version identity key)` encoded as lowercase
  hexadecimal.
- `artifact_sha256` is the lowercase SHA-256 digest of the downloaded artifact
  bytes.

The raw bytes are initially written to quarantine/temp storage and atomically
moved to the final content-addressed URI after verification succeeds. Hash
mismatches remain quarantined and are referenced by evidence records, not sent to
analyzers.

## Derived artifact paths

Derived outputs use stable artifact IDs and never raw names:

```text
fs://artifacts/manifests/{artifact_id}.json
fs://artifacts/snapshots/{artifact_id}.json
fs://artifacts/diffs/{from_artifact_id}/{to_artifact_id}.json
fs://artifacts/analyzer-output/{artifact_id}/{run_id}.json
fs://artifacts/reports/{report_id}.json
```

`artifact_id`, `run_id`, and `report_id` are controlled identifiers generated by
Tallow. They are validated as opaque safe IDs before becoming URI components.
Diff paths include both artifact IDs to avoid collisions and to make replayed
comparisons addressable.

## Retention modes

Tallow supports three retention modes for artifact bytes and derived evidence:

- `metadata_only`: store registry metadata, hashes, verification outcomes, and
  derived evidence references; discard verified raw bytes after processing.
- `suspicious_only`: retain raw bytes only for mismatches, policy failures,
  suspicious analyzer findings, or reviewer-pinned artifacts.
- `full_archive`: retain all verified raw artifacts and derived outputs subject
  to configured lifecycle limits.

Regardless of retention mode, canonical digests and evidence references remain in
PostgreSQL so observations are replayable and auditable.

## Filesystem and S3 semantics

Filesystem storage resolves `fs://artifacts/...` beneath the configured storage
root. The resolver MUST verify that the final cleaned path remains under that
root. Future S3-compatible storage will map the same URI path to an object key
under the configured bucket/prefix; object keys use the same sanitized components
and deterministic ordering as filesystem paths.

## Required tests

Storage builder tests must cover:

- deterministic URI generation for identical identity/digest inputs;
- no collisions across npm tarballs, PyPI sdists, and PyPI wheels;
- rejection of traversal, absolute path, path separator, and control-character
  inputs;
- rejection of malformed digest values;
- verification that callers use builder functions rather than hand-concatenating
  artifact paths.
